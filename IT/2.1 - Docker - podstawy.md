| Polecenie | Opis działania | Przykład / Wskazówka |

## 🧹 Czyszczenie i Sprzątanie

| Polecenie | Opis działania | Przykład / Wskazówka |
| :--- | :--- | :--- |
| `docker rm <id_lub_nazwa>` | Usuwa zatrzymany kontener z systemu. | Kontener musi być najpierw zatrzymany przez `docker stop`. |
| `docker rmi <nazwa_obrazu>` | Usuwa pobrany obraz z dysku lokalnego (zwalnia miejsce). | `docker rmi nginx` |
| `docker system prune` | **Wielkie sprzątanie:** usuwa wszystkie zatrzymane kontenery, sieci i wiszące (dangling) obrazy. | Dodaj `-a` (`docker system prune -a`), aby usunąć też nieużywane obrazy. |

> [!tip] Wskazówka dotycząca ID kontenerów
> Nie musisz kopiować ani przepisywać całego, długiego identyfikatora kontenera (np. `4a7b8c9d1e2f`). Dockerowi wystarczą pierwsze **3-4 znaki** (np. `docker stop 4a7b`), o ile jednoznacznie identyfikują dany kontener na liście!

## 🛠️ Mechanizm "wchodzenia" do wnętrza uruchomionego kontenera.
```bash
docker exec -it <id_lub_nazwa_kontenera> sh
```

| Składnik | Pełna nazwa | Co dokładnie robi? |
| --- | --- | --- |
| **`docker`** | CLI | Uruchamia główny program Dockera. |
| **`exec`** | *execute* | Nakazuje wykonanie nowego polecenia **wewnątrz już działającego** kontenera. |
| **`-i`** | *interactive* | Trzyma otwarte wejście (STDIN) – Docker słucha Twojej klawiatury. |
| **`-t`** | *tty* | Przydziela wirtualny terminal – odpowiada za znak zachęty (`/#`), kolory i klawisz Tab. |
| **`<id>`** | Adresat | Wskazuje konkretny kontener (np. `e9f1`), w którym odpalamy komendę. |
| **`sh`** | *shell* | Podstawowy, lekki interpreter poleceń Linuxa, który posłuży nam za terminal. |

> [!tip] Skrót myślowy
> Połączone flagi **`-it`** oznaczają po prostu: *"chcę wejść w interakcję z kontenerem za pomocą normalnego, działającego terminala"*. Komenda `sh` jest używana zamiast `bash`, ponieważ znajduje się w absolutnie każdym, nawet najbardziej odchudzonym obrazie Linuxa.

## 🎬 Docker Compose & Deweloperstwo (Automatyzacja)

Gdy projekt staje się większy (np. wymaga aplikacji i osobnej bazy danych), zamiast pojedynczych komend `docker run`, używamy **Docker Compose** i pliku konfiguracyjnego `docker-compose.yml`.

```bash
docker compose up -d    # Uruchamia cały projekt w tle
docker compose watch    # Uruchamia tryb automatycznego odświeżania kodu

```

### 📊 Narzędzia Deweloperskie (Hot-Reload)

| Polecenie | Opis działania | Przykład / Zastosowanie |
| --- | --- | --- |
| `docker compose watch` | **Tryb "Obserwatora":** Monitoruje pliki na Twoim komputerze i automatycznie synchronizuje zmiany z wnętrzem kontenera w czasie rzeczywistym. | Uruchamiasz w osobnym terminalu podczas pisania kodu. Zmieniasz kod w IDE -> aplikacja w Dockerze natychmiast ma nowe pliki bez restartu. |
| `docker compose up --build` | Wymusza ponowne przebudowanie obrazów przed uruchomieniem projektu. | Przydatne, gdy dodasz nową bibliotekę do projektu (np. do pliku `requirements.txt`). |
| `docker compose down` | Zatrzymuje i całkowicie usuwa kontenery, sieci i domyślne zasoby stworzone przez `up`. | Czyści środowisko po zakończonej pracy nad projektem. |

> [!abstract] Różnica: `docker run` vs `docker compose`
> * `docker run` – Zarządzanie **jednym, surowym kontenerem** z poziomu CLI (dobre na szybkie testy, np. `docker run nginx`).
> * `docker compose` – Orkiestracja **całego projektu** na podstawie pliku `docker-compose.yml`. Pamięta za Ciebie porty, zmienne środowiskowe i zależności między kontenerami.
